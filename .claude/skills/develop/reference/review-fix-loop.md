# 修正 & 再レビュールール

Phase 4 のレビュー後、指摘があった場合にこのファイルを参照する。

## 基本原則

IMPORTANT: レビュー指摘に基づいてコードを修正した場合、修正の規模・重要度にかかわらず必ず再レビューを通す。「MEDIUM だから再レビュー不要」という自己判断は NG。

## 指摘の分類

レビュー指摘を以下の3つに分類する:

| 分類 | 対応 | 注意 |
|------|------|------|
| **修正する** | `implementer` agent にレビュー結果を渡して修正 | HIGH/MEDIUM/LOW すべて修正対象。重要度が低いことは無視する理由にならない |
| **スコープ外** | `/issue` スキルを発火して Issue を作成し、番号を記録 | `gh issue create` を直接叩かない。口だけで「別 Issue で」と言って作らないのは NG |
| **対応不要** | 副作用の内容を具体的に明記してスキップ | 修正すると副作用が出る場合 **のみ**。「軽微だから」「LOW だから」は対応不要の理由にならない |

## 修正→再レビューループ

1. 修正があった場合、再度 `/review` スキルを発火して再レビュー
2. **ループは最大3回。** 3回目のレビュー後の残存指摘は以下で処理:
   - **HIGH が残存**: ワークフロー中断。ユーザーに報告して判断を仰ぐ
   - **MEDIUM が残存**: 必ずユーザーに報告し、`/issue` スキルを発火して Issue 化する
   - **LOW が残存**: 内容に応じて動的に判断。ユーザーに報告のみで Issue 化は必須ではない
3. 修正が一切なかった場合のみ再レビューをスキップして Phase 6 に進む

## Phase 6 遷移前チェック

IMPORTANT: 以下が全て未達の場合、Phase 6 に進んではならない。

- [ ] 「スコープ外」に分類した指摘は全て `/issue` スキルで Issue を作成済みか？
- [ ] 3回目レビュー後の残存 MEDIUM は全て `/issue` スキルで Issue を作成済みか？
- [ ] 作成した Issue 番号を控えているか？（PR のレビューサマリーで `#番号` として記載する）
